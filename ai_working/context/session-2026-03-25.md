## What was accomplished

### FR-8.9: Australian Employment Data Integration (full feature)

Eight-phase implementation adding Australian workforce data to the Tier 1 industry intelligence pipeline.

#### Phase 1: Alembic migration 014
- Added `region` column to `industry_occupation_profiles` (default `'US'`)
- Updated unique constraint to `(soc_code, naics_code, region)`
- Created `abs_employment` table (raw ABS/JSA occupation-industry data)
- Created `anzsco_soc_concordance` table (ANZSCO→SOC semantic matches)

#### Phase 2: NAICS↔ANZSIC crosswalk
- Loaded 21 mappings into `industry_crosswalk` table
- Manual concordance via ISIC Rev.4 bridge (20 NAICS sectors → 19 ANZSIC divisions)
- One split mapping: NAICS 51 "Information" → 85% ANZSIC J / 15% ANZSIC S

#### Phase 3: ANZSCO→SOC semantic matching
- Embedded 4,694 ANZSCO title variants (491 unit groups) using all-MiniLM-L6-v2
- Queried existing 66,512 O*NET title embeddings via pgvector cosine similarity
- Results: 346 auto-accepted (>=0.85), 56 needs review (0.70-0.85), 89 low confidence (<0.70)
- Low confidence entries predominantly ABS statistical categories (not real occupations)

#### Phase 4: ABS employment ingestion
- Ingested 2,743 rows from JSA Occupation Profiles
- 21.2M total workers; 17.3M allocated via 50/30/20 rank-weighted distribution
- ~18% unallocated employment (outside top 3 industries per occupation)

#### Phase 5: AU industry profiles computation
- Generated 1,084 `industry_occupation_profiles` rows with `region='AU'`
- Joined ABS employment → ANZSCO-SOC concordance → Eloundou scores
- Exposure scores identical to US (occupation-level, not region-specific)

#### Phase 6: API region parameter
- All 4 sector endpoints accept optional `?region=AU` parameter
- `GET /api/v1/sectors`, `GET /api/v1/sectors/{code}`, `GET /api/v1/sectors/{code}/priorities`, `GET /api/v1/sectors/composite`
- Default `region=US` — fully backward compatible
- Refactored `sector_priorities.py` to compute location quotient from profiles table (single code path for both regions)

#### Phase 7: Frontend RegionSelector + page updates

### Bug fix: Case-insensitive region parameter (commit e9dcc29)
- Backend regex pattern `^(US|AU)$` rejected lowercase `us`/`au` with a 422 error
- Changed to `^(US|AU|us|au)$` with `.upper()` normalisation in all 4 sector endpoints
- Root cause of AU data not displaying was a separate issue: the backend server required a manual restart after adding the region WHERE clauses — Windows uvicorn `--reload` did not detect the changed files

### Bug fix: MS/AEI score joins for AU profiles — 6-digit vs 8-digit SOC (commit 68ef8ae)
- After the initial FR-8.9 implementation, MS AI applicability and AEI exposure columns were NULL on all 1,084 AU profiles (0% coverage)
- Root cause: `anzsco_soc_concordance` stores 8-digit SOC codes (e.g. `29-1141.00`) sourced from `onet_occupations`; Microsoft and AEI tables use 6-digit codes (e.g. `29-1141`)
- US profiles were unaffected because all US source tables (oews_employment, ms_ai, aei) use 6-digit codes consistently
- Fix: changed the AU profiles computation join to use `SUBSTRING(onet_soc FROM 1 FOR 7)` prefix match instead of exact equality
- Coverage after fix: Microsoft 0% → 92%, AEI 0% → 91%
- The platform now has three SOC code formats in play — see ADR-004 Decision 7 for the full architectural record

## Key architectural decisions

See ADR-004 for full details:
1. Region column on existing table (not separate AU table)
2. ANZSCO→SOC via semantic matching (not manual curation)
3. Employment distribution via 50/30/20 rank-weighted split
4. Location quotient from profiles table (single code path)
5. URL-based region state (not React context)
6. Two-hop industry crosswalk at sector level (NAICS→ISIC→ANZSIC)
7. SOC code format normalisation — SUBSTRING prefix match for AU MS/AEI joins (8-digit vs 6-digit)

## Known issues / next steps

- **50/30/20 approximation** — acceptable for Tier 1; Tier 2 AU engagements would need exact ABS cross-tabulation
- **56 ANZSCO mappings need review** — in the 0.70-0.85 confidence band; can be reviewed manually if higher accuracy needed
- **NZ/UK expansion** — same pattern (add region rows, build concordance, ingest employment) would work without schema changes
- **SOC code format inconsistency** — three formats coexist across tables: 8-digit O*NET (`29-1141.00`), 6-digit MS/AEI/OEWS (`29-1141`), 7-char prefix join key. AU joins use SUBSTRING prefix match; US joins use exact equality. See ADR-004 Decision 7. Not yet standardised at ingestion time.
- **Windows uvicorn --reload caveat** — on Windows, uvicorn `--reload` does not reliably detect newly created or modified `.py` files in the `app/` tree. After adding new route handlers or service functions, a manual server restart is required to pick up the changes. This caused AU data to silently serve US data during FR-8.9 development until the server was restarted.
- **Tier 2** features (FR-1 Org Hierarchy → FR-7 Privacy → FR-6 Dashboards) are next in the build chain
